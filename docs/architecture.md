## 🎯 아키텍처 목표

CafeMatch 앱의 아키텍처는 다음을 목표로 설계되었습니다:

✅ **확장성** - 기능 추가가 쉬운 구조  
✅ **유지보수성** - 코드 이해와 수정이 간단  
✅ **테스트 가능성** - 각 부분을 독립적으로 테스트

✅ **보안** - 사용자 데이터 보호

## 🏢 전체 구조 이해하기

CafeMatch는 **3가지 주요 계층**으로 구성됩니다:

### **1. 화면 계층 (Presentation Layer)**
> 사용자가 보고 만지는 부분

- **역할:** 사용자 인터페이스 표시
- **포함:** 지도 화면, 리뷰 화면, 프로필 화면 등
- **상태 관리:** 화면의 상태를 관리 (로딩 중? 에러? 데이터 표시?)

**예시:**
```
홈 화면을 열면:
1. "로딩 중..." 표시
2. 데이터 받으면 지도에 마커 표시
3. 마커 클릭 → 카페 상세 정보 표시
```

### **2. 비즈니스 로직 계층 (Domain Layer)**
> 앱이 **무엇을 해야 하는지** 정의

- **역할:** 앱의 핵심 규칙과 기능 로직 정의
- **포함:** 카페 데이터 모델, 리뷰 검색 로직, 팔로우 기능 등

**예시:**
```
"리뷰 작성 규칙":
- 별점은 1-5점 사이
- 텍스트는 최소 10자, 최대 500자
- 사진은 최대 5장
```

### **3. 데이터 계층 (Data Layer)**
> 데이터를 **어디에서** 가져오는지 정의

- **역할:** 데이터를 가져오고, 저장하고, 변환
- **포함:** Firebase에서 데이터 가져오기, 지도 API 호출, 로컬 저장소 관리

**예시:**
```
카페 검색할 때:
1. 네이버 API에 요청 → 카페 목록 가져오기
2. 로컬 저장소에 저장 (나중에 오프라인에서 쓸 수 있도록)
3. Firebase에서 리뷰와 평점 가져오기
```

```
    ↓

    응답 받음 → 파싱 → 저장

    ↓
    화면 계층
    지도에 마커 표시
    ↓
    사용자
    "지도에 카페가 표시됨"
```

## 🎮 핵심 구성 요소 이해하기

### **1. Firebase (백엔드 서비스)**

**Firebase란?** Google에서 제공하는 백엔드 서비스 (클라우드 기반)

**역할:**
- 🔐 **인증:** 사용자 로그인/회원가입
- 🗄️ **데이터베이스 (Firestore):** 리뷰, 프로필 저장
- 📸 **저장소 (Storage):** 사진 저장
- 🔔 **알림:** 팔로우한 사람의 활동 알림

**CafeMatch에서의 사용:**
```
사용자 로그인
    ↓ Firebase Auth
유효한 사용자 확인
    ↓
데이터베이스 접근 권한 획득
    ↓ Firestore
리뷰, 프로필, 팔로우 정보 읽기/쓰기
```

### **2. 지도 & 검색 API**

#### **Google Maps API**
- 역할: 지도 표시, 마커 표시, 길 찾기
- 장점: 정확도 높음, 기능 풍부

#### **Naver Place API**
- 역할: 카페와 식당 정보 검색
- 장점: 한국 데이터 풍부, 실시간 정보

**데이터 흐름:**
```
Naver API → 카페/식당 검색 → 위도/경도 얻기
                              ↓
                    Google Maps에 마커 표시
```

### **3. 상태 관리 (Provider)**

**상태란?** 화면에 표시해야 할 현재 정보

**예시:**
- 로딩 중인가? (로딩 바 표시)
- 데이터가 있는가? (리스트 표시)
- 에러가 발생했는가? (에러 메시지 표시)

**Provider의 역할:**
```
상태 변경 감지 → 화면에 자동으로 반영

예: 리뷰 작성 완료
1. 상태 → "리뷰 업로드 완료"로 변경
2. Provider가 감지
3. 화면이 자동으로 업데이트 → "리뷰 등록되었습니다" 표시
```

### **4. 로컬 저장소 (Hive)**

**역할:** 기기 내부에 데이터 저장 (인터넷 없을 때 사용)

**사용 사례:**
```
1. 카페 목록 검색 후 저장
2. 인터넷 끔
3. 앱 다시 열면 이전 검색 결과 표시 가능 ✅
```

## 📊 계층별 책임 분리

### **화면 계층이 해야 할 일**
✅ UI 표시  
✅ 사용자 입력 받기  
✅ 상태 관리  
✅ 에러 메시지 표시

❌ API 호출하지 않기  
❌ 데이터베이스 접근 안 하기  
❌ 비즈니스 로직 포함 안 하기

**예시 (좋은 코드):**
```dart
// ✅ 화면은 화면에만 집중
class ReviewScreen extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return Consumer<ReviewProvider>(
      builder: (context, provider, _) {
        if (provider.isLoading) {
          return LoadingWidget();  // 로딩 표시
        }
        return ReviewListWidget(provider.reviews);  // 리뷰 표시
      },
    );
  }
}
```

### **비즈니스 로직이 해야 할 일**
✅ 규칙 정의 (리뷰는 최소 10자)  
✅ 데이터 검증  
✅ 기능 로직 (팔로우 기능이 뭘 해야 하는지)

❌ UI 코드 안 쓰기  
❌ 직접 API 호출 안 하기  
❌ 특정 프레임워크에 의존하지 않기

**예시:**
```dart
// ✅ 비즈니스 로직
class CreateReviewUseCase {
  Future<void> execute(Review review) async {
    // 규칙 검증
    if (review.text.length < 10) {
      throw ValidationException('최소 10자 이상');
    }
    
    // 데이터 검증 후 저장
    await repository.save(review);
  }
}
```

### **데이터 계층이 해야 할 일**
✅ API 호출  
✅ 데이터베이스 조작  
✅ 로컬 저장소 관리

❌ 비즈니스 로직 포함 안 하기  
❌ 화면 정보 알지 않기

## 🔐 보안 구조

```
사용자 데이터 보호 방법:

1. 인증 (Authentication)
   → 본인이 맞는지 확인 (로그인)

2. 권한 (Authorization)
   → 본인의 데이터만 접근 가능
   예: 내 리뷰는 나만 수정 가능

3. 암호화
   → 데이터를 암호화해서 저장
   예: 비밀번호는 텍스트로 저장 안 함

4. HTTPS
   → 인터넷 통신 암호화
   예: 카페 API 호출도 암호화된 통신
```

**Firebase Rules 예시:**
```
규칙: 모든 사용자가 모든 리뷰를 볼 수 있다 ✅
하지만: 자신의 리뷰만 수정/삭제 가능 ✅

이렇게 제한함으로써 보안 유지
```

## ⚡ 성능 최적화 전략

### **1. 이미지 캐싱**
```
첫 번째 카페 방문:
1. 인터넷에서 사진 다운로드
2. 기기에 저장

두 번째 방문:
1. 저장된 사진 바로 표시 (빠름!) ⚡
```

### **2. 지도 마커 최적화**
```
카페가 500개 이상일 때:

나쁜 방법:
500개 마커 모두 지도에 표시
→ 매우 느림 ❌

좋은 방법:
마커 클러스터링 (가까운 마커끼리 묶기)
→ 1개 그룹으로 표시 → 빠름 ✅
→ 클릭하면 펼침
```

### **3. 데이터베이스 쿼리 최적화**
```
나쁜 방법:
모든 리뷰 로드 (1000개+ 로드)
→ 느림 ❌

좋은 방법:
최신 리뷰 10개만 로드 (페이지네이션)
→ 빠름 ✅
→ 스크롤하면 다음 10개 로드
```

## 🔄 개발 과정에서 아키텍처의 역할

### **1단계: 새 기능 추가**
```
기능: "카페를 즐겨찾기에 추가" 기능 만들기

Step 1: 데이터 계층
└─ Firebase에 저장 방법 정의

Step 2: 비즈니스 로직
└─ "즐겨찾기 추가" 규칙 정의

Step 3: 화면 계층
└─ 버튼과 UI 만들기

Step 4: 테스트
└─ 각 계층 독립적으로 테스트 가능!
```

### **2단계: 버그 수정**
```
버그: "리뷰 이미지가 안 업로드됨"

아키텍처 덕분에:
- 화면 부분은 괜찮음 (UI는 정상)
- 데이터 계층에서만 찾으면 됨 (Firebase 업로드)
→ 빠르게 찾아서 수정 가능!
```

### **3단계: 기능 변경**
```
변경: "Naver API" → "Google Maps API" 사용

아키텍처 덕분에:
- 데이터 계층만 바꾸면 됨
- 화면, 비즈니스 로직은 그대로 ✅
→ 쉽고 안전하게 변경 가능!
```
